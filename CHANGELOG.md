# Changelog

All notable changes to StormGuide are documented here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

`tools/Bump.ps1 -Version <x.y.z>` rolls the `Unreleased` section into a dated
heading and stamps the same version into `StormGuide/StormGuide.csproj`. New
entries between releases go under `## [Unreleased]`.

### Fixed

- Side panel rendered an empty window with no tab content on current Against the Storm builds. Root cause: AtS now ships the new Unity Input System with the legacy `UnityEngine.Input` class disabled, so every direct `Input.GetKey` / `Input.GetKeyDown` call threw `InvalidOperationException`. The throw inside `DrawWindow`'s trailing Shift-overlay check propagated out of the `GUILayout.Window` callback every frame, which is the documented Unity behaviour that aborts the window content while leaving the chrome visible. `UI/SidePanel.cs` now routes the four legacy-input call sites (Ctrl-modifier check + per-digit Ctrl+1\u20269 quick-jump, F5 catalog reload, Shift-held debug overlay) through new `SafeInputGetKey` / `SafeInputGetKeyDown` helpers that try/catch on first use, log a single warning, and short-circuit subsequent calls. The `F8` toggle keeps working because it goes through BepInEx's `KeyboardShortcut.IsDown()` path, which the input system can't disable.
- `UI/SidePanel` MonoBehaviour was silently dropping its Unity lifecycle (`Start` / `Update` / `OnGUI`) under the deployed BepInEx + r2modman profile because it was hosted on a freshly-spawned child `GameObject`. `StormGuidePlugin.SpawnPanel` now adds `SidePanel` directly to the plugin's own `BaseUnityPlugin` GameObject (whose lifecycle is confirmed live by BepInEx), `DontDestroyOnLoad`s that GameObject, and `OnDestroy` only destroys the component, not the host. Also adds one-shot `Update()` / `OnGUI()` lifecycle log lines on `StormGuidePlugin` plus a small `StormGuide alive \u2014 hold F8 to toggle full panel` corner overlay so the plugin's MonoBehaviour can be visually confirmed in-game.
- `tools/SmokeRun.ps1` failed when launched from a fresh shell with `dotnet build` reporting "No .NET SDKs were found" because the previous candidate list missed the versioned scoop layout (`scoop\apps\dotnet-sdk\<ver>\dotnet.exe`) and silently fell back to the runtime-only `C:\Program Files\dotnet\dotnet.exe`. The script now globs every versioned scoop install, requires an adjacent `sdk\` folder before accepting a candidate (so a runtime-only install can no longer hijack the build), pins `DOTNET_ROOT` to the chosen SDK directory, and invokes the resolved exe explicitly via `& $dotnet build`.

### Added

- Home tab subsection collapse/expand: every Home section (Pinned, Marked recipes, Fuel, Village, Trade, Idle, Rebalance, Risks, Needs, Orders, Glades, Cornerstones) now renders a â–ľ / â–¸ caret next to its header. Click to collapse the section body; the chosen state persists across sessions via the new `PluginConfig.HomeCollapsedSections` config string. Header decoration ("open \u203a" jump buttons, dynamic styling for fuel runway colour) stays intact when the section is expanded.
- `StormGuide/UI/SidePanel.BeginHomeSection` helper: opens a `BeginHorizontal` row with the caret + label, returning whether the section is expanded so callers can early-return without rendering the body. Persists collapsed-set diffs immediately on toggle.

ETA uses settlement-wide net flow as the denominator; clicking jumps to the Orders tab. Soonest-ETA wins when multiple orders target the same good.
- `SidePanel.PopulateRecipeOrderEtaCache` / `DrawRecipeOrderEta` helpers: populate a per-frame map of `producedGood \u2192 (order, ETA minutes)` at the top of `DrawBuildingDetail` so each recipe card looks up in O(1) instead of re-walking active orders + `MatchedGoodFor`.
- Home Trade section: a horizontal trader timeline mini-bar split into three zones \u2014 current trader's travel section (sky blue), current trader's visit window (solid green when in village, dim green when not yet arrived), and a dim slot for the next trader. A 3 px white "now" marker shows progress: positioned by exact travel-progress when en route, centered in the visit window once arrived. Header line above the bar names both traders + status. Hover tooltips on each zone explain the segments. Renders right under the section header so the player can read the rotation at a glance without scrolling.
- `SidePanel.DrawHomeTraderTimeline(cur, nxt)` helper draws the strip; visit-duration timing isn't exposed by the game, so the visit zone uses a fixed proportional width \u2014 it's a glanceable indicator, not a quantitative scale.
- Orders tab pick options gain a `\u2665 best for me` badge on the live pick whose `RewardCategories` overlap most with the player's owned-cornerstone amplifiers (e.g. a reputation-amplifying cornerstone makes a reputation-shaped pick light up). The Orders tab header now includes a `\u2665 your cornerstones amplify: X, Y` summary line so the player understands why a pick was flagged.
- `StormGuide/Domain/CornerstoneAmplification.cs` \u2014 pure heuristic mapping owned cornerstone display name + description to the set of `LiveGameState.OrderRewardEntry.Category` values they amplify (substring scan over a curated keyword table covering `reputation`/`fame`/`gold star`/`repute`, `cornerstone`/`blueprint`/`perk pick`, `resolve`/`morale`/`happiness`, `trade good`/`stockpile`/`production`/`yield`/`output`). 12 unit tests cover empty/null input, single-category matches, mixed multi-category aggregation, case insensitivity, name-only and description-only triggers, and no-keyword fallback.
- Good tab price-history chart gains a one-line trend extrapolation under the sparkline. When at least 3 samples have been collected this visit, the panel fits a least-squares line over the per-second sell-price samples, surfaces the slope in `currency/min`, and projects the regression line forward by ~3 minutes (`\u2197 trend: +X.XXX/min \u00b7 if it holds, price in ~3m \u2248 Y/u`). A flat line (|slope|<0.005/min) renders as `\u2192 trend: flat at N.NN/u this visit` so the UI never invents a non-trend; projections are clamped to non-negative prices on strong downtrends.
- `StormGuide/Domain/PriceTrend.cs` \u2014 pure linear-regression helper exposing `PriceTrend.Compute(samples, secondsBetweenSamples)` returning `(SlopePerMin, FittedNow, LastValue, SampleCount)` plus a `Project(estimate, minutesAhead)` extrapolation. Returns null for fewer than 3 samples, non-positive spacing, or a degenerate regression denominator. 12 unit tests cover null/short input guards, flat/rising/falling/noisy slope cases, spacing scaling, fitted-now alignment with the regression line at the last sample, projection forwards/backwards, and null-estimate projection.
- Villagers tab gains a per-race housing match indicator under the dietary "needs supplied" block. Reports one of `\u2713 housing: N {race-specific} House built` (with optional `+M shelter` and `\u26a0 K homeless` tags), `\u26a0 housing: shelter only (M) \u00b7 no {race-specific} House built`, or `\u2716 housing: none built ({race-specific} House needed)`. Lit green when preferred housing covers the race, amber when only shelter is up, red when nothing is built. Skipped silently when the catalog has no race-specific housing entry and no homeless are present.
- `StormGuide/Domain/HousingMatch.cs` \u2014 pure heuristic mapping a race's needs list + built-building count map + alive/homeless to a `HousingMatchResult(Level, PreferredCount, ShelterCount, PreferredHouseDisplayName, Homeless)`. Identifies the race's preferred housing root from the canonical `"X Housing"` need entry (skipping the generic `"Any Housing"` placeholder), then matches catalog `Category == "Housing"` buildings whose name starts with `root + " House"` (with `"Purged "` prefix stripped so purged variants count). 12 xunit tests cover null catalog, preferred/shelter-only/no-housing/unknown classification, purged variant counting, plural-vs-singular root matching (Foxes \u2192 Fox House), case-insensitivity, the preferred-wins tiebreak, homeless surfacing across levels, and root-matching avoiding spurious substring hits.
- `LiveGameState.BuiltBuildingCounts()` \u2014 returns `IReadOnlyDictionary<string, int>` mapping every built building's catalog model name to its instance count by walking `BuildingsService.Buildings`. Defensive against null services + missing model names; used by the Villagers tab housing indicator (and reusable for any future feature that needs a built-instance join key).
- Draft tab cornerstone options gain an inline `\u21bb stacks with: A [tag], B [tag1, tag2]` line that names every currently-owned cornerstone whose `usabilityTags` overlap with the option's tags. Capped to 4 entries with a `\u2026 and N more` tail; silent when the option has no usability tags or no owned cornerstone shares one. Renders in the green-ok style so it reads as a synergy hint distinct from the existing red-warn anti-synergy line.
- `StormGuide/Domain/CornerstoneOverlap.cs` \u2014 pure helper exposing `Compute(optionTags, ownedWithTags)` that returns `IReadOnlyList<OwnedTagOverlap>` (each with the owned cornerstone's display name + the deduplicated, case-insensitive list of shared tags). Filters out owned entries with zero overlap so the renderer doesn't have to. 12 xunit tests cover null inputs, empty option-tag lists, no-overlap, single/multi tag overlap, multi-owned filtering to the overlapping subset, case-insensitive matching, duplicate tag collapsing, empty/null tag entries skipping, blank display name skipping, and null-tags-list owned skipping.
- `LiveGameState.OwnedCornerstonesWithTags()` \u2014 returns `IReadOnlyList<OwnedCornerstoneWithTags>` joining each currently-active cornerstone's id + display name with the names of its `usabilityTags`. Used by `CornerstoneDraftProvider` to populate `CornerstoneOption.OwnedOverlap` once per pick instead of re-querying the model service per option.
- `Domain.CornerstoneOption` extended with an `OwnedOverlap: IReadOnlyList<OwnedTagOverlap>?` slot, populated by the draft provider via `CornerstoneOverlap.Compute` so the UI renders directly without re-walking the game's effect model.
- Glades tab gains a scout-driven clear-time estimator. Under the discovered/total glades summary, the panel renders `\u23f1 ~Xm to clear N glade(s) at S scout(s) (90s/scout/glade heuristic)` whenever there are remaining undiscovered glades and at least one Resource Gathering worker is assigned. With no scouts assigned, it degrades to a warn-styled `\u23f1 N glade(s) left to discover \u2014 assign Resource Gathering workers to scout` so the player still gets an actionable hint. Skipped silently when every glade is already discovered.
- `StormGuide/Domain/GladeClearTime.cs` \u2014 pure heuristic exposing `Estimate(remainingGlades, scoutCount, secondsPerScoutPerGlade = 90)` returning a `GladeClearTimeEstimate(RemainingGlades, ScoutCount, SecondsPerScoutPerGlade, MinutesToClear)` record. Returns null on non-positive remaining/scouts/rate so the UI can branch on a single guard. 11 xunit tests cover null-on-zero/negative remaining, null-on-zero/negative scouts, null-on-non-positive rate, default-rate normal case, single scout/glade, many scouts scaling down, custom-rate parameter, large-numbers stability, and the 90s default constant.
- `LiveGameState.AssignedWorkersInCategory(catalog, category)` \u2014 returns the total assigned worker count across every built `BuildingsService.Buildings` instance whose catalog model name belongs to the requested category. Defensive against null catalog/empty category/missing service; used as the scout proxy for the Glades clear-time estimator (default category: "Resource Gathering") and reusable for any future feature that needs a per-category worker tally.
- Villagers tab gains a per-race dietary variety summary directly under the existing "needs supplied" block. Renders `\U0001f374 dietary variety: A/B needs in stock (S%)` where A is the count of the race's catalog needs whose stockpile is currently > 0 and S is the rounded percentage of needs covered. Lit green at \u226580%, amber at \u226550%, red below. When at least one need is missing, an indented `missing: x, y, z` line names up to three of the empty needs with a `\u2026 (+N more)` tail. Skipped silently when the race has no needs or the stockpile lookup yields nothing meaningful.
- `StormGuide/Domain/DietaryVariety.cs` \u2014 pure helper exposing `Compute(needs, stockLookup)` returning a `DietaryVarietyResult(TotalNeeds, SuppliedCount, ScorePercent, MissingNeeds)` record. Skips empty/whitespace need entries so a malformed catalog row doesn't drag the score down, treats throwing or negative stock lookups as missing, and rounds the percent away-from-zero so 1/3 reads 33% and 2/3 reads 67%. Returns null on null inputs or zero non-empty needs. 12 xunit tests cover null/empty/whitespace inputs, all-supplied/none-supplied/partial cases, rounding tiebreaks, whitespace-skip non-dilution, throwing-lookup defensiveness, negative-stock handling, and missing-needs preserving input order.

## [1.0.0] - 2026-04-28

First public release. The `0.0.1` baseline already shipped every feature surface end-users see; `1.0.0` is what closes out the productionization work behind it: release plumbing (manifest template, CI, release workflow, version-bump script), a Domain-only test project (60 tests), centralised cache TTLs and per-section frame-cost percentiles, productionized Embark and Diagnostics tabs, and a localization passthrough adapter so non-English players can pick up translated catalog strings once the live text-service is wired.

### Added

- `CHANGELOG.md` (this file).
- `tools/manifest.template.json` â€” checked-in Thunderstore manifest template; `tools/Pack.ps1` reads it instead of building the manifest inline.
- `tools/Bump.ps1` â€” single command to bump the csproj `<Version>` and roll the changelog `Unreleased` section into a dated entry.
- `.github/workflows/ci.yml` â€” validation pipeline on PRs and `main` (catalog JSON parse, manifest template parse, `tools/CatalogTrim` build, `dotnet test` on `tests/StormGuide.Tests`, PowerShell syntax check).
- `.github/workflows/release.yml` â€” on `v*` tag, extracts release notes from this changelog and creates the GitHub Release. Asset upload is done from a workstation via `tools/Pack.ps1 -Publish` because the plugin DLL cannot be built on the runner without the game's reference assemblies.
- `tests/StormGuide.Tests/` â€” xunit project (`net10.0`) source-sharing the `Domain/` layer via `<Compile Include>`. - `StormGuide/Domain/PerfRing.cs` â€” bounded ring of frame-cost samples with `P50` / `P95` / `Percentile(p)`. Pure (game-free), so it's unit-tested in `tests/StormGuide.Tests/`.

## [0.0.1] - 2026-04-26

Baseline. Everything shipped to `main` before the changelog existed is rolled up here so the trail starts somewhere coherent.

### Added
